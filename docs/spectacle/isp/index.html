ISP

Send invoice

PUT /invoices/{invoiceId}

This endpoint adds the provided invoice to our system. To submit invoices into our system this endpoint has to be used. We will accept any invoice as long as the body is well formed JSON. Any validation errors of the invoice are picked up by workers in the background and available with GET:/invoices/{invoiceId}.

Idempotency

The endpoint is idempotent. If an invoice with the same unique identifiers (organisation number and invoice reference) is submitted and the invoice exists already in the system, it is rejected. Invoices cannot be changed.

invoiceIn

A single invoice.

invoiceId: string
in path

The invoiceId must be constructed as orgno-no.{issuerOrgno}.{invoiceRef} where {invoiceRef} is a URL-safe reference that is unique for each issuer. Maximum length of the invoiceId is 200 bytes (i.e. 200 characters if the invoice id is in ascii).

Authorization: string bearer <access_token>
in header

Authorization token is obtained from Access Token API: https://github.com/vippsas/vipps-accesstoken-api

ISP

Revoke invoice

PUT /invoices/{invoiceId}/status/revoked

Revoke an invoice. A revoked invoice is not shown to the recipient. Invoices can be revoked if they currently are in the states created, pending or rejected. If an invoice has been approved, deleted or expired it cannot be revoked anymore.

invoiceId: string
in path

The unique invoice id.

If-Match: string
in header

Property for optimistic concurrency control. Expects the etag of the last known invoice status. Status updates are only performed if the provided etag matches with the etag of the current status.

Idempotency-Key: string
in header

Unique identifier for the requested state transition, provided by the caller. This idempotency key is essential to guarantee idempotent behaviour. Do not use the same idempotency key for multiple transactions.

Authorization: string bearer <access_token>
in header

Authorization token is obtained from Access Token API: https://github.com/vippsas/vipps-accesstoken-api
